§ 0 · Preface

写在前面为什么从 nanochat 入门 LLM

如果 nanoGPT 是"语言模型的 ResNet-18"，nanochat 就是"语言模型的全链路最小可运行系统"。

nanochat 不是又一个开源大模型项目，而是 Karpathy 把"如何造一个 ChatGPT"这件事完整地、教学式地装进了一个仓库。它在 8000 行左右的 Python / Rust 代码里把整个链路打通：从 BPE 分词器开始，到在 FineWeb 上预训练一个 Llama 风格的 decoder-only Transformer，再到中训练、指令微调、（可选）强化学习，最后是一个跑得起来的 FastAPI 聊天前端。

这条链路上的每一段，都对应着工业级 LLM 的一个真实工程子系统。这意味着你读完之后获得的不是某个模型的特定知识，而是整条流水线的可迁移直觉。

CV ↔ NLP 类比

如果你做过 ImageNet 时代的 ResNet 训练：那时你也走过一条完整链路——准备 ImageNet，定义 ResNet 骨干，写训练循环跑 SGD，再做下游任务微调。nanochat 让你把这整套直觉原样平移到语言模型，并在新增的几个语言专属步骤（tokenizer / SFT / RL）上扩展认知。

§ 1 · World View

语言模型与 LLM 的世界观从 P(token | context) 开始

在写下任何 Transformer 公式之前，先建立一个最小的概率心智模型。

所有现代大语言模型，无论叫 GPT 还是 Llama 还是 nanochat，本质上都在学同一件事：给定一段 token 序列，预测下一个 token 的概率分布。听上去几乎是把一个超大号 n-gram 模型用神经网络重写了，但当你把"上下文"做到足够长、网络做到足够大、数据做到足够多之后，这件看起来简单的事就变成了"知识的隐式压缩"。

1.1什么是语言模型

设一段 token 序列为 $x_1, x_2, \ldots, x_T$，一个语言模型要学的就是联合概率 $p(x_1, \ldots, x_T)$。利用概率的链式法则，这件事可以写成：

$$ p(x_1, \ldots, x_T) \;=\; \prod_{t=1}^{T} p(x_t \mid x_{<t}) $$

也就是说，整段文本的概率 = 每一步在给定前面所有 token 的条件下，预测当前 token 的概率连乘。神经网络要做的，就是把右边这一项 $p(x_t \mid x_{<t})$ 用一个参数化函数 $f_\theta$ 来逼近。

训练时的损失函数也就自然而然出来了：把所有训练序列的 log-likelihood 加起来取负，等价于在每个位置做一个分类问题，类别数等于词表大小 $|V|$，标签是下一个 token：

$$ \mathcal{L}(\theta) \;=\; -\sum_{t=1}^{T} \log p_\theta(x_t \mid x_{<t}) $$

这就是交叉熵，在 PyTorch 里就是一行 F.cross_entropy(logits, targets)。

CV ↔ NLP 类比

ImageNet 上做分类时，对每张图像，模型输出 1000 个类别的概率，你用 cross-entropy 训。语言模型唯一的差别是：每个序列位置都是一次分类，类别数从 1000 变成了 ~50k 到 ~100k（词表大小）。每条 sequence 的有效"训练样本数"等于其长度。

1.2自回归生成（autoregressive generation）

训练时的目标是"预测下一个 token"，那么推理时怎么生成长文本？很自然地——把模型自己预测出来的 token 加回到上下文里，再让它预测下一个：

  1. 输入 prompt：[hello, world]
  2. 模型给出 next-token 分布；从中采样得到 ,
  3. 序列变为 [hello, world, ,]；继续预测下一个
  4. 采样得到 how；序列变长一格
  5. 循环直到遇到 <|eos|> 或达到最大长度

这个"边生成边把自己输出再喂回去"的过程，就叫 autoregressive decoding。它带来两个重要后果：

  • 训练可以并行（一次 forward 同时拿到所有位置的 loss），但推理本质上是顺序的——下一个 token 必须等上一个 token 生成完。
  • 因为推理时每多生成一个 token，上下文就长一格，朴素实现会重复计算前缀的注意力。这就引出 KV cache（见 §10）。

1.3从 CV 到 NLP 的关键差异

对一个 CV 背景的读者，下面这张对照表也许比任何一段散文都更有用。

有几个差异非常关键，值得多说两句：

1. 输入是离散的。 这是 NLP 和 CV 最根本的分歧。像素是连续值，可以直接做 BN / 卷积；token 是符号，必须先经过一个 embedding lookup，把整数变成 D 维向量，才能进入神经网络。整个 token 表（"我们认识哪些子词"）是模型的一部分，与模型一起训练。

2. 监督信号是免费的。 ImageNet 时代你得花钱标 1000 类标签；语言模型预训练完全不需要人工标注——只要给一段文本，"下一个词是什么"这个标签就是文本自己。这就是为什么 LLM 可以吃下整个互联网的数据，而 CV 不行。

3. 一个 forward 拿很多 loss。 在训练时，一段长度为 T 的序列经过一次 forward，可以同时拿到 T 个位置的 cross-entropy loss——它们共享了大量计算（除了最后的 head）。这让 LLM 训练的"每 FLOP 学到的信息量"对比 CV 高得离谱。

1.4ChatGPT 风格的五段式 pipeline

nanochat 完整复刻了如今大模型实践的标准链路。可以把它想象成一个串行的工厂：

每个阶段的本质：

  • Tokenize：把字符流转成 token id 流。一次性训完 BPE，之后整个 pipeline 都复用这一个 tokenizer。
  • Pretrain：在海量无标签 web 文本上做下一词预测，让模型变成一个"什么都懂一点"的通用基模型 (base model)。
  • Mid-training：在少量但有结构的数据上继续训，例如多轮对话格式 / 多选题 / 含工具调用的数学题，让模型学会"格式与角色"。
  • SFT：在精挑细选的高质量指令-回答对上做有监督微调，让模型学会"按助手的方式回答"。
  • RL：用强化学习进一步把模型的行为推向某个可衡量的目标（在 nanochat 里是 GSM8K 上的答对率）。

有趣的是，"训练目标"在 1～4 阶段其实是同一个 cross-entropy，只是数据变了。直到 RL 才换成另一种目标。这一点在 §8 会再次强调。

读源码定位

整个 pipeline 的"指挥棒"是仓库根目录的 speedrun.sh：它按顺序调用 scripts/tok_train.py、scripts/base_train.py、scripts/mid_train.py、scripts/chat_sft.py、scripts/chat_rl.py、scripts/chat_eval.py，最后启动 scripts/chat_web.py。读完本手稿后，从 speedrun.sh 倒推任何一步都不难。

§ 2 · Tokenization

分词器：从字符到 BPE把人类语言切成模型能吃的最小单位

语言模型的"第零层"，决定了模型看到的世界。

在做任何语言建模之前，我们必须先回答一个朴素的问题：模型的"输入单位"是什么？字符？单词？还是别的？这个选择听起来像工程细节，却深刻影响了模型的能力上限、词表大小、序列长度，乃至训练成本。nanochat 像几乎所有现代 LLM 一样，选了一个折中方案：子词级 BPE。

2.1为什么不能直接喂字符或词

设想两种极端的分词方式：

方案 A：字符级。 词表只有几百个字符 (a-z, A-Z, 标点, 中文常用字…)，规则简单，覆盖所有可能。但代价是序列变得极长——"transformer" 这一个英文词就要占 11 个 token。Transformer 的注意力是 O(T²)，序列每变长一倍计算量变四倍，字符级让训练几乎不可行。

方案 B：词级。 把每个英文单词当一个 token，"hello" 就是一个 id。序列短，效率高，但词表会膨胀到几十万——"running"、"runs"、"ran" 都是不同 id，模型学不到它们的内在关联。更糟的是，遇到训练时没见过的词 (OOV, out-of-vocabulary)，模型只能丢一个 <unk>。

方案 C：子词级 (subword)。 折中之道。常见词保留为单个 token ("the")，罕见词被切成若干高频子词的组合 ("unbelievable" → "un" + "believ" + "able")。这样：

  • 词表规模可控（典型 32k ~ 128k）。
  • 序列长度合理（一个英文词平均 ~1.3 个 token）。
  • 不会有 OOV——最坏情况下退化到单字符。
CV ↔ NLP 类比

ViT 把图像切成 16×16 的 patch，每个 patch 经过一个线性层变成一个 D 维向量、形成一个 token 序列。子词分词在做几乎相同的事：把字符流切成"语言里的 patch"，每个 patch 再去查 embedding 得到向量。区别在于 NLP 的 patch 大小不固定，且 patch 是通过数据驱动统计得到的。

2.2BPE：字节对编码的算法直觉

BPE（Byte Pair Encoding）的算法非常朴素，几乎是儿童拼图：反复合并语料里出现频率最高的相邻符号对。下面是一个 toy 例子。假设语料只有以下三个词（每个字符之间留空，词尾加 _）：

语料 / corpusBPE 演示
l o w _      l o w _      l o w _      l o w _      l o w _
l o w e r _  l o w e r _
n e w e s t _ n e w e s t _ n e w e s t _ n e w e s t _ n e w e s t _ n e w e s t _
w i d e s t _ w i d e s t _ w i d e s t _

第一步：统计所有相邻字符对的出现频率。("e","s") 出现了 9 次（在 newest/widest 里），是最高频的。把它合并成一个新符号 "es"，加入词表，重新切分整个语料。

第二步：在新的语料里再次找最高频对。这次是 ("es","t") → "est"。

第三步：("est","_") → "est_"，因为很多词以 -est 结尾。

这样重复 N 步，词表大小就长成 (初始字符数) + N。每一步的"合并规则"按顺序记录下来。编码新文本时，按规则顺序把字符流合并；解码时按 token id 查表拼接。

nanochat 用的是字节级 BPE（byte-level BPE），即初始 alphabet 是 256 个字节，而不是字符——这样可以无损处理任何 UTF-8 输入，包括中文、emoji、奇怪符号，根本没有 OOV 的概念。GPT-2 起这就是事实标准。

2.3nanochat 的 rustbpe

nanochat 把 BPE 训练这一步用 Rust 写了出来，放在 rustbpe/ 目录。原因很直接：训练 BPE 是个 CPU 密集型任务，Python 太慢。一旦训完，导出来的合并规则与词表 (tokenizer.pkl / json) 就可以用任何语言加载。

nanochat speedrun 默认词表大小约 65,536（即 2¹⁶）。这个数字不是凭空选的——它正好是一个 16-bit 无符号整数能表示的范围，对内存对齐和某些底层 kernel 友好。同时它也大到能让常见词独立成 token、小到不让 embedding 矩阵失控。

训练好的 tokenizer 暴露两个核心接口：

nanochat/tokenizer.py · 概念性接口python
class Tokenizer:
    def encode(self, text: str) -> list[int]:
        # "Hello world" -> [15496, 995]
        ...

    def decode(self, ids: list[int]) -> str:
        # [15496, 995] -> "Hello world"
        ...

    # 特殊 token，用于对话模板等
    bos_token_id: int
    eos_token_id: int
    user_token_id: int
    assistant_token_id: int

除了普通的合并产生的子词，分词器还需要预留一些特殊 token，用来标记对话的开始、结束、用户角色、助手角色、工具调用等。它们不参与 BPE 合并，而是手工塞进词表，分配独立的 id。这些特殊 token 就是 chat template 的基础，到 §7 / §8 还会再见到它们。

一个被分词后的文本看上去就是这样：

原文："The quick brown fox jumps."

The464 quick2068 brown7586 fox21831 jumps18045 .13

注意：空格通常和后面的词合并在同一个 token 里——这是 GPT-2 风格 BPE 的惯例。token id 仅为示意，nanochat 实际数字会不同。

注意

很多 LLM "失败案例"都源于 tokenizer：模型不会"数字母" (因为字母被合并了)；中文/代码的切分质量决定了模型在这两个域的上限；某些数字被切得乱七八糟，使算术题异常困难。Tokenizer 不只是工程细节，它限定了模型能感知到的最小语义单位。

读源码定位

rustbpe/ 整个目录是用 Rust 写的训练实现；nanochat/tokenizer.py 提供 Python 侧 wrapper；scripts/tok_train.py 是训练入口。读一遍 tok_train.py 足以理解整个流程。

§ 3 · Embeddings

嵌入：把离散符号变成连续向量从 token id 到表示空间

分词器把世界变成 id；嵌入层把 id 变回"有几何"的对象。

假设我们已经把句子变成了一串 id：[464, 2068, 7586, 21831, 18045, 13]。但 Transformer 是连续函数，不能直接吃整数。中间需要一座桥——这座桥就是 embedding 层。

3.1嵌入层就是一张查找表

本质上，embedding 层是一个形状为 (|V|, D) 的可训练矩阵 $E$。给定一个 token id $i$，它返回的就是矩阵的第 $i$ 行：

$$ \text{embed}(i) \;=\; E_i \;\in\; \mathbb{R}^{D} $$

在 PyTorch 里就是一行 nn.Embedding(vocab_size, n_embd)。前向计算等价于 one-hot 乘以 $E$，但实现上当然不会真去做这个稠密乘法，而是直接 index。$E$ 与 Transformer 的其它参数一起被 SGD/AdamW 更新。

CV ↔ NLP 类比

把 embedding 层想象成 CNN 的"第一层 + ReLU"。在 CV 里，第一层卷积把原始 RGB 变成几十个通道的特征图；在 NLP 里，embedding 把 token id 变成 D 维特征。区别在于第一层卷积"看了局部像素"，而 embedding "只看一个 id" ——上下文信息是后面的注意力层去抽取的。

3.2学到的空间结构

训练好的 embedding 矩阵会展现出令人惊讶的几何结构。Word2Vec 时代的经典发现是：

$$ E_{\text{king}} - E_{\text{man}} + E_{\text{woman}} \;\approx\; E_{\text{queen}} $$

这表明嵌入空间里的"方向"承载了语义关系（这里是性别方向）。现代 LLM 的 embedding 维度通常 D = 768 ~ 12288，能编码远比 Word2Vec 复杂的结构——但核心直觉仍然是：相似的 token 在嵌入空间里位置相近，语义关系对应着方向。

3.3nanochat 的嵌入选择

nanochat 在 GPT 类内部用了一个 nn.Embedding(vocab_size, n_embd) 来做 token embedding，并采用了一个常见小技巧：tied input/output embedding——把输入嵌入矩阵和最后预测层的权重共享 (或者强制共享转置)。这样：

  • 参数减少一份 (~ vocab_size × D)，对小模型来说节省 10-20%。
  • 逻辑上一致：把 id 映射到向量、和把向量映射回 id 用的是同一张"字典"。

nanochat 没有显式的"绝对位置嵌入" (absolute positional embedding) ——它把位置信息延后到注意力层用 RoPE 注入，这一点 §4.4 详细说。

读源码定位

nanochat/gpt.py 中 GPT.__init__ 里的 self.wte = nn.Embedding(...)（wte = "word token embedding"，沿用了 GPT-2 的命名）。

§ 4 · Transformer 核心

Transformer 核心从直觉到数学，看穿这台机器

如果你只读一章，就读这一章。所有其它内容都是它的应用。

从 2017 年的 "Attention Is All You Need" 到 2025 年的所有大模型，骨干都是 Transformer。但 nanochat 用的不是原论文那种 encoder-decoder 结构，而是当代 LLM 的标准选型：decoder-only Transformer，加 RoPE、RMSNorm、SwiGLU——可以把它看作 Llama 系列的简化版。本章会把这些零件拆开来讲清楚。

4.1整体视角：一台"信息精炼"流水线

从最高层往下看，一个 decoder-only Transformer 长这样：

  1. 输入序列 [id_1, id_2, ..., id_T] 经过 embedding 得到 $X \in \mathbb{R}^{T \times D}$。
  2. $X$ 依次经过 $N$ 个 Transformer block，每个 block 由"注意力 + FFN"两个子层组成，每个子层都带残差连接与归一化。
  3. 最后用一个 LM head（线性层）把每个位置的 $D$ 维向量映射到 $|V|$ 维 logits。
  4. 对每个位置取 cross-entropy 与"下一个 token"对照，求和取均值，得到 loss。

4.2自注意力（self-attention）：本章的灵魂

注意力机制的核心隐喻：每个 token 都是一名"读者"，它需要从序列里其它 token 那里"读"信息回来。读什么、读多少，由它自己决定。我们用三组学到的投影来形式化这件事：

  • $Q = X W_Q$（query，"我想要查询什么"）
  • $K = X W_K$（key，"我能被怎样的 query 匹配到"）
  • $V = X W_V$（value，"如果被匹配到，我会贡献什么内容"）

$X \in \mathbb{R}^{T \times D}$，$W_Q, W_K, W_V \in \mathbb{R}^{D \times d}$，于是 $Q, K, V \in \mathbb{R}^{T \times d}$。注意力的输出就是：

$$ \text{Attention}(Q, K, V) \;=\; \text{softmax}\!\left( \frac{QK^\top}{\sqrt{d}} \right) V $$

逐步解读这个式子：

  1. $QK^\top \in \mathbb{R}^{T \times T}$：第 $i$ 行第 $j$ 列等于 token $i$ 的 query 和 token $j$ 的 key 的点积——衡量 $i$ "想从" $j$ 那里读信息的强度。
  2. 除以 $\sqrt{d}$：缩放因子，防止点积随维度 $d$ 增大而方差爆炸，softmax 进入饱和。
  3. softmax 沿"j 这一维"做归一化：每个 token 对所有源头的注意力权重加起来等于 1。
  4. 乘以 $V$：用这套权重对 value 加权求和，得到每个位置的新表示。

整个操作可以读成一句话："对每个 token，把所有其它 token 的 value 按 (query·key) 的相似度做加权平均，得到它的新表示。"

注意力机制 = 内容寻址的 key-value 字典 + 可微分的加权聚合。
CV ↔ NLP 类比

卷积是"固定模板下的加权聚合"——3×3 卷积核给每个邻居一组学到的、与位置绑定的权重，作用于所有图像。注意力是"动态生成模板下的加权聚合"——权重 $\alpha_{ij}$ 由 query 和 key 在当下样本里实时算出，作用于这一具体序列。这就是注意力比卷积更灵活也更费算力的原因。

4.3Multi-Head Attention：多个视角并行

用一组 $W_Q, W_K, W_V$ 只能学到一种"查询风格"。MultiHead 的想法是同时用 $h$ 组独立的 $(W_Q^{(i)}, W_K^{(i)}, W_V^{(i)})$，每一组都在更小的维度 $d_h = D / h$ 上做自己的注意力，最后把 $h$ 个输出拼起来再投影回 $D$ 维。

$$ \text{MultiHead}(X) = \text{Concat}(h_1, \ldots, h_H) W_O, \qquad h_i = \text{Attention}(X W_Q^{(i)}, X W_K^{(i)}, X W_V^{(i)}) $$

实现上当然不会真去搞 $h$ 个独立矩阵——而是把 $W_Q, W_K, W_V$ 直接做成 $D \to D$，然后 reshape 成 $(B, T, H, d_h)$，再做注意力。代码层面上 multi-head 几乎和 single-head 一样简洁。

nanochat 还可能用了一个叫 GQA (Grouped-Query Attention) 的变体——多个 Q head 共享同一组 K/V head，能在几乎不损害质量的情况下大幅减少 KV cache 的内存占用。这对小模型省内存很有用。

4.4位置编码：RoPE

注意力机制本身对 token 的顺序是不敏感的——把输入打乱再算注意力，得到的输出也只是被同样地打乱。这显然不对，"狗咬人"和"人咬狗"差别巨大。所以我们必须把位置信息显式注入。

原始 Transformer 用的是绝对位置嵌入：给每个位置 $t$ 一个独立的 $D$ 维向量 $P_t$，与 token embedding 相加。简单但有问题——它对外推到训练时没见过的长度不友好。

nanochat 用的是 RoPE (Rotary Position Embedding)，目前是开源 LLM 的事实标准。RoPE 的思路精妙：

  1. 不再"加"位置向量到 embedding 上，而是把位置信息编码成对 Q, K 的旋转。
  2. 对每一对维度 (2k, 2k+1)，把它们看作复平面上的一个二维向量，按角度 $\theta_t = t \cdot \omega_k$ 旋转，其中 $\omega_k = 10000^{-2k/d}$ 是该维度的"频率"。
  3. 关键性质：两个被旋转过的 query 和 key 做点积时，结果只依赖于相对位置 $t_q - t_k$，不依赖绝对位置。

$$ \langle \text{RoPE}_{t_q}(q), \text{RoPE}_{t_k}(k) \rangle \;=\; f(q, k, t_q - t_k) $$

这两点带来的好处：

  • 相对性：模型学到的是"距离 5 个 token 的关系"，而不是"绝对位置 173"。语义自然。
  • 外推性：训练时见过最长 2048，推理时可以适度推到 4096 而不立刻崩。配合一些 trick (NTK-aware、YaRN) 可以推得更远。
  • 实现简洁：只在 attention 之前对 Q 和 K 做一次"旋转"操作即可，不动 V，不动其它任何东西。
注意

RoPE 不是"加在 embedding 上"的位置嵌入。它只对每个 attention 层的 Q 和 K 起作用。模型的 embedding 矩阵不包含位置信息——这是为什么 §3 里我们看到 nanochat 没有显式的 positional embedding 层。

4.5FFN / SwiGLU：每个 token 的独立深加工

注意力子层让 token 之间互相"通信"，FFN 子层让每个 token 在自己的位置上做深加工。它就是一个对每个位置独立应用的 MLP：

$$ \text{FFN}(x) \;=\; W_2 \cdot \sigma\!\left( W_1 \cdot x \right) $$

原始 Transformer 用 ReLU，后来主流换成了 GeLU。现代 LLM（包括 nanochat）通常用 SwiGLU——它本质上是一个"双线性 + 门控"的结构：

$$ \text{SwiGLU}(x) \;=\; W_2 \cdot \big( \text{SiLU}(W_g \cdot x) \;\odot\; (W_1 \cdot x) \big) $$

这里 $\odot$ 是逐元素相乘，$\text{SiLU}(x) = x \cdot \sigma(x)$ 是 swish 激活函数。SwiGLU 比 ReLU/GeLU 在等参条件下普遍涨点 0.5～1.5 个 perplexity。代价是多了一份 $W_g$ 矩阵，参数量略增——所以 hidden 维度通常会从 4D 降到 ~2.67D 来保持参数相当。

4.6RMSNorm：LayerNorm 的极简变体

原始 Transformer 用 LayerNorm：

$$ \text{LN}(x) \;=\; \frac{x - \mu}{\sqrt{\sigma^2 + \epsilon}} \cdot \gamma + \beta $$

需要计算均值 $\mu$、方差 $\sigma^2$，并维护可学的缩放 $\gamma$ 和偏移 $\beta$。RMSNorm 把它简化为只用 RMS（root mean square）做缩放，去掉均值与偏移：

$$ \text{RMSNorm}(x) \;=\; \frac{x}{\sqrt{\frac{1}{D}\sum_i x_i^2 + \epsilon}} \cdot \gamma $$

实证上，RMSNorm 在 LLM 训练中和 LayerNorm 几乎等效，但更快、更省内存，所以成了主流。nanochat 也采用 RMSNorm。

另外一个微小但重要的选择：nanochat 用 pre-norm（在每个子层之前做 RMSNorm）而不是 post-norm。pre-norm 让残差流的尺度保持稳定，训练 deep transformer 时更稳定。

4.7残差流（residual stream）视角

把整个 Transformer 看作"一条主干道 + 沿途的若干读写者"，会让很多事情变得清晰。这种视角来自 Anthropic 的可解释性研究——把 token 在网络里的表示 $x$ 看作一条残差流：

  • 每一层的注意力子层从残差流里 read 信息（通过 Q/K/V 计算），并把结果 write back 到残差流（残差加法）。
  • 每一层的 FFN 子层同样 read-then-write。
  • 残差流的"维度"$D$ 是固定的；每个子层向其中加一个增量。

这个视角的力量在于：它把"参数学的是什么"和"信息流是什么"分开了。某个 attention head 学到的，其实是从残差流的哪个子空间读、把结果写到哪个子空间。一旦你接受这个视角，"为什么用残差连接"、"为什么 layer norm 放在 pre-norm 位置"这些工程选择就变得理所当然。

4.8因果掩码（causal mask）

语言模型在训练时不能"作弊"——预测位置 $t$ 的下一个 token 时，模型只能看 $x_1, \ldots, x_t$，不能偷看 $x_{t+1}$ 之后的东西。这个约束通过因果掩码实现：在算 softmax 之前，把注意力矩阵右上三角的位置全置为 $-\infty$，softmax 之后这些位置的权重就变成了 0。

这一步在 PyTorch 里就是一行 scores.masked_fill(mask, -inf)；用 F.scaled_dot_product_attention(..., is_causal=True) 还能自动用 FlashAttention 优化掉 O(T²) 的内存开销。nanochat 利用了 PyTorch 的 SDPA，吃满 H100 的算力。

读源码定位

Transformer block 的实现在 nanochat/gpt.py 里的 Block 类——一个 CausalSelfAttention + 一个 MLP (SwiGLU)，两个都通过 RMSNorm + 残差加法接在一起。RoPE 的实现一般在 layers.py 中，作为 apply_rotary 类型的工具函数。

5.1代码骨架

nanochat/gpt.py · 概念骨架（简化）python
class CausalSelfAttention(nn.Module):
    def __init__(self, cfg):
        self.qkv = nn.Linear(cfg.d, 3 * cfg.d, bias=False)
        self.proj = nn.Linear(cfg.d, cfg.d, bias=False)

    def forward(self, x, freqs):
        q, k, v = self.qkv(x).chunk(3, dim=-1)
        q, k = apply_rotary(q, freqs), apply_rotary(k, freqs)
        y = F.scaled_dot_product_attention(q, k, v, is_causal=True)
        return self.proj(y)


class MLP(nn.Module):           # SwiGLU 风格
    def __init__(self, cfg):
        self.w1  = nn.Linear(cfg.d, cfg.d_ff, bias=False)
        self.wg  = nn.Linear(cfg.d, cfg.d_ff, bias=False)
        self.w2  = nn.Linear(cfg.d_ff, cfg.d, bias=False)
    def forward(self, x):
        return self.w2(F.silu(self.wg(x)) * self.w1(x))


class Block(nn.Module):
    def __init__(self, cfg):
        self.n1, self.attn = RMSNorm(cfg.d), CausalSelfAttention(cfg)
        self.n2, self.mlp  = RMSNorm(cfg.d), MLP(cfg)
    def forward(self, x, freqs):
        x = x + self.attn(self.n1(x), freqs)
        x = x + self.mlp(self.n2(x))
        return x


class GPT(nn.Module):
    def __init__(self, cfg):
        self.wte     = nn.Embedding(cfg.vocab, cfg.d)
        self.blocks  = nn.ModuleList([Block(cfg) for _ in range(cfg.n_layer)])
        self.norm_f  = RMSNorm(cfg.d)
        self.head    = nn.Linear(cfg.d, cfg.vocab, bias=False)
        self.head.weight = self.wte.weight          # tied

    def forward(self, idx, targets=None):
        x = self.wte(idx)                       # (B, T, D)
        freqs = precompute_rope(x.size(1))      # RoPE 频率表
        for blk in self.blocks:
            x = blk(x, freqs)
        x = self.norm_f(x)
        logits = self.head(x)                    # (B, T, V)
        if targets is None:
            return logits
        loss = F.cross_entropy(logits.view(-1, logits.size(-1)),
                               targets.view(-1))
        return logits, loss

这段代码差不多就是 gpt.py 的全貌（稍作概念化处理）。请把它和 §4 的概念图对照来看——每一个名字都对应一个你刚学过的零件。

5.2规模选择：depth-driven scaling

nanochat 给出了几个预设规模，通过一个核心超参 depth (= 层数 N) 来推导出所有其它维度。常见命名：d20 / d26 / d32 等，数字就是 depth。一组合理的耦合关系是：

  • $D$ (hidden size) ≈ 64 × depth，即 d20 ≈ 1280，d26 ≈ 1664，d32 ≈ 2048。
  • $H$ (num heads) ≈ depth/2 或某个让 $d_h = D / H$ 落在 64 ~ 128 的值。
  • $d_{ff}$ (FFN hidden) ≈ 4D 在 SwiGLU 风格下调成 ~2.67D。
  • 训练步数与数据量按 Chinchilla 的 token/param ≈ 20 调比。

这是一种"用一个数控制一切"的工程审美——你只需要决定"我要跑多深的模型"，其它都自动级联出合理值。这背后是大量的 scaling law 直觉。

speedrun 默认是 d20（约 5.6 亿参数），在 8×H100 上跑约 3-4 小时，云上租用 ~$100。这就是著名的 "$100 base model" 的来源。

5.3优化器：Muon + AdamW 的混搭

nanochat 一个让 CV 读者觉得新鲜的细节是它的优化器选型：

  • 对所有矩阵参数（Linear 的权重，shape ≥ 2D）：使用 Muon，一个针对矩阵更新做了特殊正交化处理的优化器。它把每一步的梯度做一个 Newton-Schulz 迭代逼近的"矩阵符号"操作，相当于沿着所有奇异值方向都用大致相同的步长更新。
  • 对其它参数（embeddings, biases, LayerNorm gain）：使用经典的 AdamW。

这是近期一两年开源社区里出现的一个有意思的训练 trick。在等算力下，Muon 比 AdamW 在矩阵参数上效率提升明显。但 Muon 的更新规则在低维参数（向量、bias）上没有意义甚至不稳，所以混搭使用。这个细节在 nanochat 的 trainer 代码里有明确体现。

CV 读者特别注意

你以前用 AdamW + cosine LR + warmup 就够了；LLM 训练有一些新工具进入主流。Muon 还在快速演进，但已经是 nanochat、modded-nanoGPT 等仓库的默认选择。理解它就要理解 Newton-Schulz iteration、矩阵正交化、谱归一化这些概念。

5.4稳定训练的几个工程要点

nanochat 为了在小预算下稳定训出，还有几个值得点名的细节：

  • bfloat16 训练：模型权重和激活在 bfloat16，loss 在 fp32 累加。混合精度是标配。
  • 梯度裁剪：torch.nn.utils.clip_grad_norm_ 把全局梯度模长裁到 1.0，防 spike。
  • warmup + cosine LR：典型 1-2% warmup，剩余按 cosine 衰减到峰值的 10%。
  • FSDP / DDP：在多 GPU 上用 PyTorch FSDP 把权重分片，让模型能放进每张 GPU。
  • DataLoader 的 shuffle：因为预训练数据是流式打包成 token，shuffle 主要在 chunk 级。

§ 6 · Pretraining

预训练让模型在互联网上读 1000 亿 token

这一步是花钱最多、决定上限最深的一步。

Pretraining 阶段做的事其实就是 §1.1 里写的目标：在大规模 web 文本上最小化 next-token cross-entropy。复杂的不是数学，而是数据准备、分布式工程与调度技巧。

6.1数据：FineWeb

nanochat 默认用 FineWeb 作为预训练数据——这是 HuggingFace 整理的、从 Common Crawl 派生出来的高质量 web 文本子集。它已经做完了去重、语言筛选、低质过滤等繁琐工序，开箱即用。

整体的数据处理流程：

  1. 下载 FineWeb 的 shards (每个 shard 一个 parquet)。
  2. 用第二章训出来的 tokenizer 把所有文本编码成 token id。
  3. 把 id 流打包成 .bin 文件（典型用 uint16，因为词表 ≤ 65536）。
  4. 训练时用 mmap 方式按 chunk 读取，零拷贝。

这种"先编码再训练"的做法是所有现代 LLM 训练的标配。tokenize 是 CPU bound 的，提前做完一次，训练时 GPU 才不会被 tokenizer 拖慢。

6.2训练目标

把一个 batch 切成长度 $T$ 的序列：

$$ \mathbf{x} = [x_1, x_2, \ldots, x_T], \quad \text{targets} = [x_2, x_3, \ldots, x_{T+1}] $$

也就是说，"输入" 和 "目标" 是同一段序列错位一格。一次 forward 拿到 $(B, T, V)$ 的 logits，与 targets 做 cross-entropy。注意每个位置都是一个独立的分类问题，所以 effective batch ≈ $B \cdot T$ 个样本。

训练主循环骨架（高度简化）：

6.4评估：bpb 与验证 loss

训练时会周期性评估两件事：

  • Val loss：在 hold-out 的 web 文本上算 cross-entropy。这是 pretraining 期最直接的指标。
  • bpb (bits per byte)：把 cross-entropy 换算到"平均每个字节用多少 bit 编码"，便于跨 tokenizer 比较。bpb 越低越好，好模型在 web 文本上一般 ~0.7-1.0 bpb。

注意：pretraining 期间的 loss 几乎单调下降，但和"模型有多聪明"并不是线性关系。下游能力（MMLU, GSM8K）的浮现往往非常突变，几乎需要"够多 token、够大模型"才会出现。

§ 7 · Mid-training

中训练学会"它现在是个聊天的助手"

数据格式从"web 流水"换成"对话剧本"。

Pretraining 出来的模型是个 base model：它会续写任何文本，但它不知道自己是个 chatbot。你问它 "What is 2 + 2?"，它可能续写出"is a math problem often used in elementary school..." 而不是"4"。让模型从"续写任意网页"切换到"扮演对话助手"，这就是 mid-training 与 SFT 要做的事。

7.1为什么需要"中间"一步

从 pretrain 直接跳到 SFT 也可以，但效果会差。原因是 base model 还没"见过"对话格式——它不认识 <|user|> / <|assistant|> 这些特殊 token 该长什么样、该出现在哪里。把对话格式与一些有结构的能力数据（多选题、数学题、工具调用）混在一起做中训练，相当于在 base model 与 SFT 之间架了一座过渡桥。

7.2对话模板（chat template）

nanochat 使用一套与 GPT-4 / Claude 等商用模型类似的 chat template。一段对话会被序列化成这样：

chat template · 概念示例format
<|bos|>
<|user|>What is the capital of France?<|eou|>
<|assistant|>The capital of France is Paris.<|eoa|>
<|user|>And of Germany?<|eou|>
<|assistant|>The capital of Germany is Berlin.<|eoa|>
<|eos|>

这些角标都是特殊 token，由 tokenizer 显式预留 id。它们的作用：

  • 告诉模型每一段是谁说的（用户 / 助手 / 系统 / 工具）。
  • 给模型"何时停"的信号——预测出 <|eoa|> 后就把这一轮 assistant turn 结束。
  • 区分训练时哪段算 loss——典型只对 assistant 那一段算 cross-entropy，user 的部分作为上下文不计 loss。

7.3数据混合

nanochat 的 mid-training 数据是一个有意思的混合体：

  • 多轮对话 (SmolTalk 子集) ——学对话格式。
  • 多选题数据 (MMLU-style) ——学"读题目+在 A/B/C/D 里选"的格式。
  • GSM8K 数学题，含工具调用 ——学"什么时候调 calculator"。
  • 少量 base-model 风格的 web 文本 ——防止灾难性遗忘。

这个阶段的训练目标仍然是同一个 next-token cross-entropy，只是 loss 通常在 user 段被 mask 掉，只对 assistant + special token 那部分算 loss。

读源码定位

scripts/mid_train.py 是入口；nanochat/chat.py / tasks/ 目录里有 chat template 的格式化与对话数据加载逻辑。

§ 8 · SFT

监督微调"换数据，不换损失"

SFT 让模型从"会聊天的格式"长成"会好好回答的助手"。

SFT (Supervised Fine-Tuning) 是从 base/mid model 走向真正 chatbot 的关键一步。它惊人地朴素：用同一个 next-token cross-entropy，在精挑细选的指令-回答数据上继续训几个 epoch。复杂性不在算法，而在数据质量。

8.1本质：还是 cross-entropy

SFT 在数学上和 pretraining 几乎没差别：

$$ \mathcal{L}_{\text{SFT}}(\theta) \;=\; -\sum_{(q, a) \in \mathcal{D}_{\text{SFT}}} \sum_{t \in \text{assistant turns}} \log p_\theta(a_t \mid q, a_{<t}) $$

$\mathcal{D}_{\text{SFT}}$ 是高质量对话数据集，loss 只在 assistant 段累加。换了数据，没换损失。这是 SFT 最重要的 mental model——它不是"另一种训练方式"，而是 pretraining 的延续，只是数据从"什么都吃"变成了"只吃好的"。

8.2nanochat 的 SFT 数据

nanochat 用的是 SmolTalk 的子集，单轮居多。SmolTalk 由 HuggingFace 整理，覆盖范围广、风格干净，是开源界做小规模 SFT 的常用选择。除此之外可能还混入一些显式的安全模板与小样本数学/代码样例。

8.3训练细节

  • 训练步数远小于 pretraining（典型 1-3 个 epoch）。
  • 学习率比 pretrain 末期还要低，避免抹掉预训练里学到的世界知识。
  • batch size 通常更小，因为单条样本更短。
  • 只对 assistant token 算 loss，user token 与 special token 之外的内容 mask 掉。
陷阱

SFT 之后的模型有强烈倾向去"模仿"训练数据里的语气和长度——如果 SFT 数据全是 5 句话以内的简短回答，模型就再也学不会写长文档了。这是 SFT 的根本局限——它只能模仿你展示给它的东西。要让模型超越训练数据，需要 RL（下一章）。

读源码定位

scripts/chat_sft.py 是入口；和 mid_train 共享大部分数据 loader。

§ 9 · Reinforcement Learning

强化学习 / GRPO第一次"换损失函数"

SFT 的极限是模仿；RL 让模型超越示例去优化目标。

前面所有阶段都在做"让 next-token 分布逼近某个数据分布"。RL 是 nanochat pipeline 里第一次让模型去优化一个外部定义的标量目标。它在 nanochat 里被用得很克制——只在 GSM8K (数学题) 上做，因为那是少数几个有客观"对/错"奖励的任务。

9.1为什么 SFT 不够

SFT 只能教模型"在训练数据展示过的回答风格里挑一个最像的"。但很多任务的最优答案是高度组合性的：数学题里有 1000 种正确的推理路径、代码题里有无数种通过测试的写法。光靠模仿，模型永远学不到"这道题怎么解"，只能学到"这道题看起来该长啥样"。

RL 的提议很简单：让模型自己尝试生成答案，对每个答案给一个 reward（数学题就是"答对了 = 1，答错 = 0"），然后用 policy gradient 把能拿到高 reward 的输出概率提高，低 reward 的降下来。

9.2GRPO：一个用得开的 RL 算法

近两年开源 LLM 里 RL 的事实标准从 PPO 慢慢演化到了 GRPO (Group Relative Policy Optimization)，由 DeepSeek 在 2024 年提出，nanochat 也采用它。核心想法分两步：

步骤一：同 prompt 多 sample。 对每个 prompt $q$，让当前模型用一定的 temperature 采样 $G$ 个回答 $\{a_1, a_2, \ldots, a_G\}$（典型 $G = 4$ 到 $16$）。每个回答 $a_i$ 都跑 reward 函数得到一个标量 $r_i$。

步骤二：用 reward 在 group 内做归一化。 计算 group 内 reward 的均值 $\mu$ 与标准差 $\sigma$，每个回答的"优势 (advantage)"为：

$$ \hat{A}_i \;=\; \frac{r_i - \mu}{\sigma + \epsilon} $$

这就把"绝对 reward"变成了"在这批 sample 里的相对好坏"。优势函数高的 sample 应该被鼓励，低的应该被压制。

步骤三：policy gradient 更新。 用一个 PPO-clip 风格的目标更新策略 $\pi_\theta$（也就是模型）：

$$ \mathcal{L}_{\text{GRPO}}(\theta) \;=\; -\frac{1}{G} \sum_{i=1}^{G} \min\!\Big( \rho_i(\theta)\, \hat{A}_i,\; \text{clip}(\rho_i(\theta), 1-\epsilon, 1+\epsilon)\, \hat{A}_i \Big) \;+\; \beta\, \text{KL}(\pi_\theta \| \pi_{\text{ref}}) $$

其中 $\rho_i(\theta) = \pi_\theta(a_i \mid q) / \pi_{\text{old}}(a_i \mid q)$ 是新旧策略的概率比，clip 防止单步更新过猛，KL 项把策略拉回参考模型 (SFT 后的模型) 附近，防止 reward hacking 时崩坏。

和 PPO 的区别

PPO 需要一个独立的 value network 估计 baseline；GRPO 用 group 内 reward 的均值作为 baseline，省掉了 value model，工程上简单很多，对小模型尤其友好。这就是为什么 GRPO 在 nanochat 这种规模的 repo 里能跑得起来。

9.3nanochat 的 RL 设定

nanochat 把 RL 限定在 GSM8K（小学数学应用题）上：

  • Reward 函数：模型输出里抽出最后的数字，和标准答案做匹配——对 = 1，错 = 0。简单、可验证、便宜。
  • Group size：每个 prompt 采样 8 个回答。
  • Reference model：取 SFT 阶段结束时的 checkpoint，固定不动。
  • 训练步数：相对 pretrain 极少（数千 step）。

RL 阶段的最大代价是采样——每一步 forward 之前都要先 generate $G$ 个回答，这个 generate 过程比 single forward 慢得多。所以 nanochat 的 RL 是可选的：speedrun 里默认跑，但你可以跳过。即使跳过，从 SFT 出来的模型也已经能用了。

读源码定位

scripts/chat_rl.py 是 RL 训练入口；nanochat/grpo.py（或 trainer 内的相关函数）实现 GRPO loss；tasks/gsm8k.py 给出 reward 计算。

§ 10 · Inference

推理与服务KV cache、采样、聊天前端

把训练好的权重变成一个真正能对话的系统。

训练只解决了"模型是什么"，剩下的"怎么把它跑起来给人用"是另一套工程。这一章讲三件事：generation loop 的本质、KV cache 为什么必不可少、采样策略的几个旋钮。

10.1朴素 generate 循环

从最朴素的 generation 开始：

generate · 朴素版本（O(T²) 累计 forward）python
def generate_naive(model, prompt_ids, max_new=200):
    ids = prompt_ids.clone()
    for _ in range(max_new):
        logits = model(ids)[:, -1, :]      # 只看最后位置
        next_id = sample(logits)
        ids = cat([ids, next_id], dim=-1)
        if next_id == eos_id: break
    return ids

问题在第三行：每多生成一个 token，整条序列都要被重新 forward 一次。第 $t$ 步的代价是 $O(t)$，整个 generate 的总代价是 $O(T^2)$——而且大部分计算是重复的：第 100 步算的前 99 个位置的注意力和第 99 步几乎一样。

10.2KV cache：把"重复"省掉

注意到一个事实：因果掩码意味着位置 $t$ 的 K, V 一旦算出来，就永远不会再变。新生成的 token 不会"反向修改"已有 token 的 K, V。所以我们可以把每一层的 K, V 缓存下来，下一步只算新位置的 Q, K, V，把新 Q 与整张缓存里的 K做注意力即可。

代价是 KV cache 占显存：每层、每 head、每个 token 都要保存一份 K 和 V 张量。对长上下文 (~128k) 模型，KV cache 可能比模型权重本身还要大。优化 KV cache (GQA, MQA, paged attention 等) 是 LLM 推理工程的核心战场之一。nanochat 是教学仓库，KV cache 的实现保持简单，但 idea 完整。

10.3采样策略：temperature, top-k, top-p

给定 logits，怎么从中"选"下一个 token？

  • Greedy：直接取 argmax。确定但单调，容易重复。
  • Pure sampling：直接按 softmax(logits) 采样。多样但容易跑偏到低概率的 token。
  • Temperature：在 softmax 前把 logits 除以 $\tau$。$\tau \to 0$ 退化为 greedy，$\tau \to \infty$ 退化为均匀分布。典型 0.7 ~ 1.0。
  • Top-k：只在概率最高的 k 个 token 里采。截断长尾。
  • Top-p (nucleus)：按概率从高到低累加，截到累积概率达到 p 为止，在这些 token 里采。比 top-k 更自适应。

nanochat 的 inference engine 把这些都暴露成参数。日常对话默认温度 ~0.7、top-p ~0.9 比较稳。要做评估 (eval) 时建议用 greedy，因为可复现。

10.4Chat Web 前端

nanochat 还配套了一个最小聊天前端：FastAPI 后端 + 简单 HTML/JS 页面 (scripts/chat_web.py)。它做的事很直接：

  1. 启动一个 FastAPI server，加载训练好的权重。
  2. 页面把用户输入按 chat template 序列化，发请求。
  3. 后端 stream-style 逐 token 返回，前端边收边渲染。

麻雀虽小五脏俱全——它让你在本地浏览器里就能和你刚训出来的模型聊天。这是整个 pipeline 最有"魔法时刻"感的一步。

读源码定位

nanochat/engine.py 实现采样/KV cache 的 Engine 类；scripts/chat_web.py 是 FastAPI server；前端模板在 nanochat/web/ 目录下。

§ 11 · Evaluation

评估如何知道你训出来的模型"有多强"

没有评估就没有迭代。

训练 LLM 和训练分类器最大的体验差异，就是评估的复杂性。ImageNet 给你一个 top-1 准确率就完事了；LLM 没有任何单一指标能覆盖它的能力面。nanochat 内置了几个标准评估集，分类如下。

11.1核心评估集

名称测什么形式
MMLU跨学科常识57 个学科的多选题 (A/B/C/D)；测"知识广度"
ARC-Easy / ARC-Challenge小学科学常识多选题；测"科学常识推理"
GSM8K小学应用题给一段题目，模型必须 chain-of-thought 解出最终数字
HumanEval代码生成给函数签名+docstring，让模型补函数体，跑单测看通过率
ChatCORE综合对话能力nanochat 自家的小规模综合评估

11.2多选题怎么"考"模型

多选题的实现技巧：把题目和选项一起编码，分别在四个选项的位置上取 logits，比较"该选项的对数概率"（或归一化后的 acc-norm）。无需让模型"输出 A/B/C/D"，更稳定也更可复现。

11.3生成类评估的麻烦

GSM8K 和 HumanEval 这类生成式评估要让模型真正生成答案。陷阱很多：

  • 解析模型最终输出的数字 / 代码，需要写规则提取。
  • 采样随机性意味着多跑几次结果会变；评估时用 greedy 才公平。
  • HumanEval 还需要在沙箱里执行模型生成的代码——nanochat 用 timeouts 简单防御一下。

11.4报告（report.md）

跑完 speedrun.sh 后，nanochat 会生成一份 report.md，把所有评估结果汇总成表格。Karpathy 在 demo 里展示过这份 report——这就是你跑出来的"$100 model"的成绩单。

读源码定位

scripts/chat_eval.py 是评估入口；tasks/ 目录下每个评估集一个文件，分别实现各自的 prompting + scoring 逻辑。

§ 12 · Running it

跑起来从读代码到亲手训一遍

最有效的学习，是真的让它在你眼前跑一次。

读懂一切之后，让它真的跑一次。下面是建议的实操路径。

12.1最小可行环境

nanochat 的"$100 speedrun"目标硬件是 8 × H100 80GB。在 RunPod / Lambda / Vast.ai 等云上租 ~$3/小时，跑 ~3-4 小时就能完成完整 pipeline。如果只有单卡：

  • 单 H100 / A100 80GB：可以跑 d20，但训得慢，几十小时。
  • 单 RTX 4090：可以跑更小的 d12 / d16，作为"它真的能跑"的验证。
  • CPU/Mac：不行，至少要能跑 CUDA。

12.2建议的分步学习实操

  1. 第一步：克隆仓库，poetry install 把环境装好。读 speedrun.sh。
  2. 第二步：单独跑 scripts/tok_train.py，训一次 BPE。打开 tokenizer 看 vocab。
  3. 第三步：用极小配置（depth=8、几千 step）跑一次 base_train.py，观察 loss 下降。
  4. 第四步：跑完整 speedrun。观察每一步生成的 checkpoint。
  5. 第五步：打开 chat_web.py，让它服务你的 SFT 模型，在浏览器里聊。
  6. 第六步：跳出 speedrun，尝试改一个东西：换数据、换 depth、关掉 RL、加 layer——观察 report.md 变化。

12.3该看的指标

  • 训练时：loss / 步、tokens/sec、GPU 利用率、显存占用。
  • 评估时：MMLU / ARC / GSM8K / HumanEval / ChatCORE 五项。
  • 主观时：自己在 chat UI 里问几个固定问题，作为"smell test"。
